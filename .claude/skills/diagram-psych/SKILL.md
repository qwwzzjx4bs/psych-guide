# Diagram Psych

**総合病院精神科医師が、外来診察中に即座に参照できる**臨床意思決定支援HTMLを生成する。

## 利用者プロファイル

| 項目 | 内容 |
|------|------|
| **職種** | 総合病院精神科医師 |
| **知識レベル** | 非常に詳しい（専門家） |
| **利用場面** | 外来診察中・病棟回診前・症例整理 |
| **目標** | 症状 → 診断（ICD-10/DSM-5-TR）→ サブタイプ → 薬物・非薬物治療の当意即妙な選択 |

---

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**exploreサブエージェント**を起動して最新の診断基準・治療エビデンスを調査する。

```
Task({
  subagent_type: "explore",
  description: "精神科疾患の最新診断・治療エビデンス調査",
  prompt: `
    以下の精神科疾患・テーマについて徹底的に調査してください：
    【対象】{疾患名またはトピック}

    調査項目：
    1. ICD-10分類コードおよびDSM-5-TR診断基準（主要症状・除外基準・罹患期間）
    2. DSM-5-TRのサブタイプ・特定用語（specifier）の一覧と定義
    3. ファーストライン薬物治療（薬剤名・用量・エビデンスレベル）
    4. セカンドライン・増強療法オプション
    5. 推奨される非薬物治療（CBT, 心理教育, 支持的療法, TMS等）とエビデンスレベル
    6. 治療抵抗例への対応
    7. 参照した主要ガイドライン名・年度

    結果は構造的に整理して返してください。
  `
})
```

### Phase 1: 臨床情報の構造化

1. 入力された症状・主訴を分析
2. 鑑別診断リストの生成
3. DSM-5-TR/ICD-10の診断ツリーを設計
4. エビデンスに基づく治療選択肢を整理
5. Phase 0のリサーチ結果と照合して抜け漏れを確認

### Phase 2: 診断フレームワーク設計

| 設計要素 | 内容 |
|---------|------|
| 主訴・症状クラスター | 入口となる症状群の整理 |
| 鑑別診断 | 除外すべき疾患の明確化 |
| 診断基準の要点 | A基準・B基準・除外基準を簡潔に |
| サブタイプ分類 | DSM-5-TR specifierによる分岐 |
| 重症度評価 | 評価スケール（PHQ-9, PANSS, YMRS等）との対応 |

### Phase 3: 治療アルゴリズム設計

| ステップ | 内容 |
|---------|------|
| Step 1 | 薬物治療ファーストライン（薬剤名・用量・エビデンスレベル） |
| Step 2 | 非薬物治療（心理社会的介入・CBT・支持的傾聴等） |
| Step 3 | 治療反応不良時の切り替え・増強 |
| Step 4 | 治療抵抗性への対応（クロザピン、ECT、TMS等） |
| 注意事項 | 禁忌・相互作用・モニタリング項目 |

### Phase 4: HTML生成

1. Tailwind CSS CDN使用
2. **Lucide icon**を使用（絵文字禁止）
3. 医療臨床カラースキーム適用
4. エビデンスレベルバッジ配置（Level A/B/C）
5. 診断ツリー・治療フローチャートを視覚化
6. 重要度・緊急度による色分け

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 5: 精度チェック（サブエージェント）

**専門家視点**で、臨床的正確性を忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "精神科図解の臨床的精度チェック",
  readonly: true,
  prompt: `
    あなたは経験豊富な精神科医師です。
    最新のDSM-5-TRおよびICD-10の知識を持ち、
    エビデンスに基づく精神科薬物療法・心理社会的療法に精通しています。

    以下のHTML図解を読んで、**臨床的精度を厳しく**チェックしてください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **診断基準の正確性**: DSM-5-TR/ICD-10の基準と齟齬はないか
    2. **サブタイプ分類の網羅性**: 重要なspecifierの抜け漏れ
    3. **薬物治療の妥当性**: 用量・ファーストライン選択の適切性
    4. **非薬物治療の網羅性**: エビデンスのある介入が漏れていないか
    5. **エビデンスレベルの正確性**: A/B/C分類が適切か
    6. **禁忌・注意事項の記載**: 重要な安全情報の欠落

    ## 出力形式

    ### 臨床的問題点（修正必須）
    - ...

    ### 改善推奨事項（追記すると有用）
    - ...

    ### 良好な点（維持すべき）
    - ...

    ### 総評
    臨床有用性: 5段階評価
    一言コメント
  `
})
```

### Phase 6: ブラッシュアップ

Phase 5のレビュー結果を踏まえてHTMLを修正する。

**優先度**:
1. 臨床的問題点 → 必ず修正
2. 改善推奨事項 → 可能な限り対応
3. 良好な点 → 維持

**よくある修正パターン**:
- 診断基準の記載精度向上
- エビデンスレベルバッジの修正
- 治療ステップの順序・内容修正
- 禁忌・注意事項ボックスの追加
- 重要な非薬物治療の追記

### Phase 7: デプロイ（任意）

```bash
# 1. 一時ディレクトリ作成
DEPLOY_DIR=$(mktemp -d)

# 2. HTMLファイルを$DEPLOY_DIRに書き込む

# 3. surge.shにデプロイ
cd "$DEPLOY_DIR" && surge . --domain {ドメイン名}.surge.sh
```

## 品質チェックリスト

### 診断精度

- [ ] ICD-10コードが正確に記載されている
- [ ] DSM-5-TR診断基準のA〜E基準が網羅されている
- [ ] 重要なサブタイプ・specifierが全て記載されている
- [ ] 鑑別診断が3つ以上示されている
- [ ] 重症度評価スケールが記載されている

### 治療内容

- [ ] ファーストライン薬物治療が用量範囲と共に記載されている
- [ ] エビデンスレベルバッジ（A/B/C）が全治療選択肢に付いている
- [ ] 非薬物治療が薬物治療と同等の比重で記載されている
- [ ] 治療抵抗例への対応が記載されている
- [ ] 禁忌・注意事項・モニタリング項目がある

### デザイン

- [ ] Lucide iconを使用（絵文字禁止）
- [ ] エビデンスレベルで色分けされている
- [ ] 診断ツリーが視覚的フローとして表現されている
- [ ] 外来中でも素早く参照できる構成（スクロール最小化）
- [ ] スマホでも読みやすいレスポンシブ対応
