サブエージェント（Task）

Agentモードで起動できる専門サブエージェント

サブエージェントとは： AgentモードのAIが、複雑なタスクを分担するためにTask()で呼び出す専門エージェント。 並列実行・読み取り専用隔離・独立したコンテキストで動作する。
並列実行のコツ：独立した調査・実装を複数の Task() に分けて同時に走らせると大幅に時間短縮できる。
generalPurpose
汎用エージェント

複雑な調査・コード検索・多ステップタスクの自律実行。ファイル編集・ツール利用が可能。

用途例：ライブラリ調査、バグ修正、機能実装
explore
読み取り専用・高速探索

コードベースの読み取り専用調査に特化。ファイル検索・パターン検索・構造把握が高速。

用途例：「どこに定義があるか調べて」「API構造を把握して」
shell
コマンド実行専門

Bashコマンド・Git操作・npm/pip・Docker等の端末操作に特化したエージェント。

用途例：git操作、ビルド、テスト実行、デプロイ
cursor-guide
Cursor製品ドキュメント専門

Cursorの機能・設定・使い方に関する公式ドキュメントを参照して回答する専門エージェント。

用途例：「Cursorのルール設定方法は？」「ホットキーを教えて」
best-of-n-runner
並列実験・隔離ブランチ実行

独立したGitワークツリー上でタスクを実行。複数の実装案を並列に試して最良を選ぶ「best-of-N」実験が可能。

用途例：複数リファクタ案の並列検討、実験的な実装の安全なテスト

Cursor 拡張機能

AIエージェントをカスタマイズ・強化する4つの仕組み

Rules（ルールファイル）
.cursor/rules/*.mdc

プロジェクト・ファイルパターン・モード別に「常時有効な指示」を設定。コーディング規約、命名規則、使用ライブラリの縛りなどを一度書けば毎回自動で適用される。

設定例
  • • 「TypeScriptでは any を使わない」
  • • 「コメントは日本語で書く」
  • • 「テストは Jest を使う」
Skills（スキルファイル）
.claude/skills/*/SKILL.md

タスク固有のノウハウ・手順をMarkdownで記述し、必要なときだけ読み込む仕組み。「このプロジェクトでHTMLを作るときはこの手順で」という専門知識をAIに注入できる。

活用例
  • • デプロイ手順の自動化スキル
  • • 特定フレームワーク向け実装ガイド
  • • ドキュメント生成の書式スキル
MCP（Model Context Protocol）
外部ツール連携プロトコル

Anthropic発のオープン規格。Cursorに外部サービスのツール・データを接続し、AIが直接操作できるようにする。DB参照・Slack投稿・GitHub操作などを1つのチャットで完結。

主なMCPサーバー例
  • • filesystem：ファイルシステム操作
  • • github：PR作成・Issue管理
  • • postgres / sqlite：DB直接クエリ
  • • slack / notion：チームツール連携
Background Agents
クラウド上での非同期実行

Cursorをバックグラウンド（クラウド）で実行するモード。IDEを閉じていても長時間タスクが継続する。「夜中に動かして朝に結果確認」といった非同期ワークフローが可能。

向いているタスク
  • • 大規模テストスイートの実行
  • • 長時間のリファクタリング作業
  • • 複数PRの並列生成
  • • CI/CDパイプラインの自動修正

よくある質問

迷ったときのQ&A・アンチパターン集

コーディング用途ならClaudeが優位です。ToolUseの安定性・長い指示への追従性・日本語対応の精度でClaudeに軍配が上がります。

画像・最新情報・マルチモーダルならGPTが有利。スクリーンショットからのUI実装、Web検索が必要なタスクはGPT-5.5やGPT-4oが向いています。

結論：メインはClaude Sonnet 4.6、画像が絡む場合はGPT-5.5、という使い分けが最もバランスが良いです。

o3 は「何度試してもClaude/GPTで解けないとき」の切り札です。具体的には：

  • 原因不明のバグで他モデルが見落とした箇所を探す
  • 数学的な最適化・アルゴリズム設計
  • 複雑な状態遷移や依存関係の整理

コスト対策：まずClaude Sonnet 4.6 → 解決しなければ o4-mini → それでも無理なら o3、という順番で試すと無駄が少ないです。

これはAgentモードのよくある落とし穴です。対策：

  • Gitを使う：変更前に git stash または作業ブランチを切っておく
  • Plan→Agent の順番：大きな変更はPlanモードで設計を確認してからAgentを起動
  • スコープを絞る：「〇〇ファイルだけ編集して」と明示する
  • 承認モードを活用：Cursor設定でファイル変更前に確認を入れる

消費を最小化する実践的なコツ：

  • Tab補完は必ず cursor-small（消費0）を使う
  • 「これって何？」程度の質問は GPT-4o mini（消費0）で十分
  • Ask モードは一部のモデルでスロー枠（無制限）を使う
  • 長い会話は途中で要約してコンテキストを圧縮する
  • コンテキストに不要なファイルを @添付しない

Claude・GPT・Gemini はいずれも日本語対応の精度が非常に高く、日本語で指示しても実用上の精度低下はほぼありません。

ただし以下の点に注意：

  • 技術用語（関数名・ライブラリ名）は英語のまま使う方が確実
  • コードコメントは Rules で言語を統一しておくと一貫性が上がる

最大の違いは「独立したコンテキスト」と「並列実行」です。

  • 通常Agentは1つの会話スレッドで逐次処理
  • サブエージェントは親Agentとは別のコンテキストで動作
  • 複数のサブエージェントを並列起動して調査・実装を同時進行できる
  • exploreなど読み取り専用タイプは誤操作のリスクがない

コンテキスト枯渇は精度低下・挙動不安定の原因になります。対策：

  • 新しいチャットを開始する：長くなったら要点をまとめて新規会話へ
  • Gemini 2.5 Pro に切り替える：2Mトークンなのでほぼ枯渇しない
  • 関係ないファイルの @添付を外す：コンテキストを圧縮
  • 「ここまでの内容を要約して」と依頼する：引き継ぎプロンプトを作ってもらう

Cursor Settings → MCP から設定できます。基本的な手順：

  1. 設定画面でMCPサーバーのJSON設定を記入
  2. サーバーを起動（多くはnpxコマンド1行）
  3. AgentモードでMCPツールが自動認識される

まず @cursor-guide サブエージェントに「MCPの設定方法を教えて」と聞くと最新の公式ドキュメントを教えてくれます。
